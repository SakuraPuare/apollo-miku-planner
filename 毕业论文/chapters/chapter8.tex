\chapter{系统集成与综合实验}
\label{chap:integration}

前三章已分别完成多因子威胁度建模、横向最优通行带构造与ST动态走廊的算法设计。本章把这些算法整合进Apollo Planning的C++代码，通过消融实验量化各组件的必要性，用Dreamview在环截图展示C++接入后的规划行为，并在全场景对比矩阵中核算轨迹质量与求解开销。

\section{与Apollo系统的集成实现}
MIKU已按Apollo的Scenario-Stage-Task架构完成Planning侧C++工程接入，改动集中在\texttt{LaneFollowPath}的路径边界构建环节，其余模块保持原状，并通过\texttt{LaneFollowPathConfig}中的独立开关在Baseline与MIKU之间切换。第\ref{sec:experiment_setup}节的连续数值指标取自Python复现实验，Dreamview截图取自Apollo工程在环运行，前者用于量化指标对照，后者用于展示C++接入后的规划行为。改进后的系统架构如图\ref{fig:sys_arch}所示。输入层沿用Apollo原有的Routing、感知障碍物、预测轨迹与车辆状态。处理层在参考线处理流水线之后新增两个以粉色标注的模块，其中威胁度评估模块输出差异化安全裕度$\delta_i$，动态权重ST边界构建模块把时空走廊以动态ST上界形式注入QP。输出层仍为Apollo标准的\texttt{ADCTrajectory}，下游Control模块无需改动。

\subsection{Task流水线中的插入位置}
MIKU作用于路径规划阶段。工程实现中，在\texttt{LaneFollowPath}路径Task内新增MIKU路径边界分支，由\texttt{GetBoundaryFromMikuObstacles}替代原\texttt{GetBoundaryFromStaticObstacles}的逐静态障碍物处理流程，将\texttt{PathBoundsDecider}中的逐障碍物独立决策改为分组全局决策。

新增的核心函数包括三项。
\begin{enumerate}
\item \texttt{GroupObstaclesByLongitudinalOverlap}\quad{}实现扫描线检测纵向重叠连通分量的逻辑，对应算法\ref{alg:optimal_band}的纵向分组阶段。

\item \texttt{CalculateOptimalNudgeDirection}\quad{}实现组内按横向排序、计算间隙、选取最大间隙并分配避让方向的逻辑，对应算法\ref{alg:optimal_band}的组内决策阶段。

\item \texttt{ProjectObstacleAtTime}\quad{}查询Prediction模块的预测轨迹，计算动态障碍物的时变SL投影，对应算法\ref{alg:optimal_band}的时变投影阶段。
\end{enumerate}

路径Task的执行流程保持不变，仍然遵循Apollo的Scenario-Stage-Task三级架构，仅在路径边界构建环节将逐障碍物贪心逻辑替换为分组全局最优逻辑。速度规划阶段的Task流水线无需新增Task，MIKU输出的\texttt{miku\_speed\_constraints\_}由\texttt{LaneFollowPath::BuildMikuSTDrivableBoundary}转换为标准\texttt{STDrivableBoundary}，并在\texttt{PiecewiseJerkSpeedOptimizer}中与原有ST边界取交集后收紧QP的$s$上界。

上述核心函数的C++接口如代码\ref{lst:interface}所示，其命名和参数风格与Apollo现有代码保持一致。

\begin{lstlisting}[caption={核心函数接口定义}, label={lst:interface}]
// 障碍物分组：基于纵向重叠的连通分量检测
std::vector<std::vector<size_t>> GroupObstaclesByLongitudinalOverlap(
    const std::vector<MikuObstacleInfo>& infos,
    bool enable_grouping);
// 组级最优避让方向计算：最大间隙策略
MikuGroupDecision CalculateOptimalNudgeDirection(
    const std::vector<MikuObstacleInfo>& infos,
    const std::vector<size_t>& group_indices,
    double road_lower,
    double road_upper,
    bool enable_max_gap);
// 时变SL边界获取：融合动态障碍物预测
bool ProjectObstacleAtTime(
    const ReferenceLine& reference_line,
    const Obstacle& obstacle,
    double time,
    SLBoundary* sl_boundary);
\end{lstlisting}

该修改策略以尽量小的代码改动量为目标。原有的\texttt{UpdatePathBoundaryBySLPolygon}函数签名和输出格式不变，Baseline分支仍可通过配置回退；MIKU分支在\texttt{PathBoundsDeciderUtil}中新增分组、排序、选间隙、分配方向和逐$s$边界更新流程。上层\texttt{LaneFollowPath::Process}仍按\texttt{DecidePathBounds}、\texttt{OptimizePath}、\texttt{AssessPath}执行；下游路径优化器接收到的仍是Apollo标准\texttt{PathBoundary}，速度优化器接收到的仍是Apollo标准\texttt{STDrivableBoundary}。

另一个重要的实现细节是，动态障碍物的纳入没有直接改动Baseline使用的\texttt{IsWithinPathDeciderScopeObstacle}静态性过滤条件，而是在MIKU分支中由\texttt{GetBoundaryFromMikuObstacles}自行遍历非虚拟、非IGNORE障碍物。这样Baseline分支仍保留Apollo原始静态障碍物处理口径，MIKU分支则可将动态障碍物通过时变SL投影纳入路径边界构建，而非被简单忽略。

\subsection{与现有LaneBorrow机制的关系}
Apollo已有的LaneBorrowPath机制用于处理长期阻塞场景。当\texttt{IsLongTermBlockingObstacle}检测到某障碍物的阻塞计数器\texttt{cycle\_counter}累积达到\BorrowMaxDepth 个规划周期，约\BorrowMaxDepth{}$\times$\PlannerCycleMs\,ms以上时，触发借道决策，允许自车借用相邻车道通行。该机制是Apollo处理本车道不可通行场景的既有方案。

MIKU与LaneBorrowPath为互补关系。
\begin{enumerate}
\item 本文方法在LaneBorrowPath之前介入，通过全局最优的避让方向分配在本车道内找到最宽的通行带，减少BLOCKED判定的发生频率。Baseline下被判定为BLOCKED的许多场景，本文方法可以找到可行的本车道通行带，无需触发LaneBorrowPath。

\item 当所有间隙均不足以容纳自车，即$W^{*} < 0$或$W^{*}$极小时，确实需要借道，此时LaneBorrowPath机制仍然发挥作用。

\item 本文方法先尝试本车道最优通行带，若不可行则触发LaneBorrowPath，LaneBorrowPath在借道车道上同样可以应用本文方法寻找最优通行带。
\end{enumerate}

工程实践中，本文方法对LaneBorrowPath的减负效果较为明显。Baseline中贪心策略产生的虚假BLOCKED频繁触发LaneBorrowPath的计时器\texttt{cycle\_counter}，自车因此频繁执行不必要的借道操作，增加了变道风险和对邻道交通的干扰。本文方法在本车道内找到全局最优解后，将LaneBorrowPath的触发限定在真正需要借道的场景，整体行驶稳定性随之提升。

\subsection{降级机制}
自动驾驶系统对安全性和鲁棒性要求严格，改进算法须具备降级机制以保证异常情况下不劣于原始系统。为此本文设计三级降级策略。
\begin{enumerate}
\item 当所有间隙均不足车宽，即$W^{*} \leq 0$时，表明在当前障碍物配置下本车道确实不存在可通行空间，此时回退到Apollo原始流程，执行\texttt{TrimPathBounds}截断路径，触发FallbackPath停车决策。这是合理的降级行为，因为BLOCKED判定是真实的而非贪心策略造成的虚假阻塞。

\item 当Prediction模块未提供某些障碍物的预测轨迹，或预测轨迹的时间范围不足以覆盖$\tau(s)$时，该障碍物退化为静态处理，使用$t = 0$时刻的位置快照。这保证了算法在信息不完整时仍能正常运行，且不劣于Baseline。

\item 若单个连通分量内的障碍物数量超过预设阈值$k_{max}$，默认取15，对该分量内部采用Baseline的逐障碍物策略作为保底。算法本身在$k = 15$时仍可正确运行，设置阈值的目的是在异常输入下提供确定性的行为保证。
\end{enumerate}

以上三种降级策略遵循``不劣于Baseline''原则。在本文测试的\MScenarioTotal{}个场景下，MIKU的输出至少与Apollo原始方案持平。降级触发时会在Debug日志中记录降级原因和上下文信息，便于后续分析和优化。

\subsection{数值仿真平台与测试场景}

数值仿真程序以Python复现Apollo Planning流水线的主要计算环节，依次模拟路径与速度规划的七个阶段，即\texttt{PathBoundsDecider}、PathOptimizer、\texttt{SpeedBoundsDecider}先验、\texttt{PathTimeHeuristicOptimizer}、\texttt{SpeedDecider}、\texttt{SpeedBoundsDecider}终验与\texttt{PiecewiseJerkSpeedOptimizer}，二次规划子问题统一采用OSQP求解。本节连续数值指标来自该Python程序，第\ref{sec:dreamview_demo}节Dreamview在环截图来自Apollo C++工程运行，两组结果分别对应指标对照与工程行为展示，不混作同一数据源。Baseline与MIKU在同一程序内并列实现，对每个场景同步输出SL路径边界、ST走廊与$v(t)$、$a_x(t)$、$a_y(t)$、$j(t)$四条时序曲线的对照结果，量化指标同步写入文件以保证可溯源与可复现。

测试场景共\MScenarioTotal{}个，由\MPressureScenarioTotal{}个原始压力场景与\MComparableScenarioTotal{}个配对可比场景构成。P1--P4保留原始临界构型，用于暴露Baseline在动态障碍物、统一安全裕度与逐障碍物贪心决策下的失效边界；C1--C4逐一复制对应压力场景，仅轻量调整时间窗或障碍物横向位置，使Baseline与MIKU均可完成通过，用于在同一可达前提下比较轨迹质量与求解开销。如此既保留了原场景对算法必要性的验证，也避免把通过与失败的离散对照直接解读为连续指标优势。

\begin{enumerate}
\item P1/C1 单行人横穿。C1在P1基础上延长规划时间窗，用于比较动态障碍物提前投影后的纵向平顺性。
\item P2/C2 行人加停车。C2在P2基础上轻量外移停车横向中心，使Baseline恢复可达，用于比较动静混合障碍物分组后的速度QP开销。
\item P3/C3 窄路加双侧交通锥。C3在P3基础上将对称锥列改为轻微非对称锥列，使Baseline可通过，同时保留差异化安全裕度释放通行空间的效果。
\item P4/C4 单车道维修封闭加借道。C4在P4基础上将维持段水马横向中心调至Baseline贪心切换临界侧，使双方均可导流通过，用于比较组级边界组织带来的求解开销差异。
\end{enumerate}

Baseline设定说明如下，Apollo原生\texttt{PathBoundsDecider}通过\texttt{IsWithinPathDeciderScopeObstacle}将动态障碍物排除在路径阶段之外，交由\texttt{SpeedBoundsDecider}在ST阶段通过时间窗约束兜底；MIKU则通过时变SL投影将动态障碍物显式纳入路径边界构建。因此P1--P4压力构型中Baseline的失败并不必然等于Apollo整套流水线的失败，而是特指“路径阶段单独处理动态障碍物的能力”这一比较口径。C1--C4可比构型在同一口径下调整时间窗或障碍物位置，使Baseline在ST阶段兜底可通行，以便在可达前提下对比MIKU与Apollo原生方案的轨迹质量与求解开销。本文所引用的效率结论仅基于C1--C4；P1--P4保留原始构型，用于说明两种策略在路径阶段对动态障碍物的敏感性差异。

\section{消融实验}

消融实验只使用P1--P4压力场景。本节定义\AblationComponentTotal{}个正交组件，据此派生\AblationVariantTotal{}个对照变体，结合通过率硬门与\ScoreDimensionCount{}维加权几何平均给出综合评分，进而在热力图、雷达图与单组件讨论中定位各组件的边际贡献。

\subsection{消融对象与变体设计}

\subsection{综合评分体系}

为把多维指标压缩为单一可比标量，本节采用两层评分架构。第一层为安全硬门，要求$\mathrm{success}=1$且未触发$\mathrm{blocked}$，任一项失败则该变体在该场景直接计零分，避免短板被高分项稀释。第二层把\ScoreMetricCountTotal{}项软指标按\ScoreDimensionCount{}个维度分组，组内子指标等权乘积，组间按权重加权几何平均，最终得到$0$至$1$区间的场景得分$S_{\mathrm{scn}}$。维度划分与权重如表\ref{tab:score_weights}所示，权重设计遵循「通行与平顺并列优先、鲁棒次之、开销最末」的车辆工程院评审口径。

\label{tab:score_weights}
\small
\begin{tabular}{llcl}
\toprule
\textbf{维度} & \textbf{权重} $w_d$ & \textbf{子指标数} & \textbf{子指标} \\

每个子指标在场景内以M5\_full作分母做相对归一化。「越大越好」类指标取$x/x_{\mathrm{ref}}$；「越小越好」类指标取$x_{\mathrm{ref}}/x$。选用M5为分母而非Baseline，是因为Baseline在多个场景中触发$\mathrm{blocked}$导致部分指标爆炸不收敛，而M5是已验证可行解，作分母数值稳定。归一化公式为
\begin{equation}
n_i^{\mathrm{larger}} = \frac{x_i + \varepsilon}{x_{\mathrm{ref}} + \varepsilon}, \qquad
n_i^{\mathrm{smaller}} = \frac{x_{\mathrm{ref}} + \varepsilon}{x_i + \varepsilon},
\label{eq:score_norm}
\end{equation}
其中$\varepsilon = 10^{-3}$用于防零除。

单场景得分由硬门系数$G \in \{0, 1\}$与\ScoreDimensionCount{}维加权几何平均共同构成
\begin{equation}
S_{\mathrm{scn}} = G \cdot \prod_{d=1}^{\ScoreDimensionCount} \left[ \prod_{i \in d} n_i^{1/|d|} \right]^{w_d},
\label{eq:score_scn}
\end{equation}
全局得分对$K=\AblationScenarioTotal{}$个压力场景再做几何平均并放大至百分制
\begin{equation}
S_{\mathrm{overall}} = \begin{cases}
\left(\displaystyle\prod_{k=1}^{K} S_{\mathrm{scn}}^{(k)}\right)^{1/K} \times 100, & \text{若 } \forall k,\; S_{\mathrm{scn}}^{(k)} > 0, \\[4pt]
\dfrac{1}{K}\displaystyle\sum_{k=1}^{K} S_{\mathrm{scn}}^{(k)} \times \AblationZeroScenarioScale, & \text{若 } \exists k,\; S_{\mathrm{scn}}^{(k)} = 0.
\end{cases}
\label{eq:score_overall}
\end{equation}
几何平均比算术平均更适合工程评分体系，因其具有否决性，单项极差直接拉低总分而非被其他高分项掩盖。当某变体在任一场景触发硬门导致$S_{\mathrm{scn}}=0$时，几何平均会退化为零分；为区分不同程度的部分失效，此时退回算术平均并折半惩罚。

上述以M5\_full为基准的相对归一化使得通过率对总分起主导作用。硬门系数$G$将未通过场景的得分直接归零，而M1、M2、M3在关键压力场景中的失败导致对应$S_{\mathrm{scn}}=0$，几何平均随之退化，全局得分骤降至\AblationFailedScoreCeil 分以下。这意味着综合得分在很大程度上反映的是「通过率差异」而非「通过场景内的四维度质量差异」。同时，M5作为归一化参考并不等同于强制其在所有子指标上占优；M4\_no\_C5与M5\_full的差值仅为\AblationMarginMFourMFive 分，属于当前压力场景下C5未触发活跃边界、计算开销近似相同造成的微小数值差异，不应解读为C5削弱规划质量。读者在解读表\ref{tab:ablation_score}的综合排名时，应同时关注图\ref{fig:ablation_heatmap}中各变体在通过场景内的逐场景明细得分，以及图\ref{fig:ablation_radar}的四维度雷达图，以获得各组件在非硬门维度上的真实贡献幅度。

\subsection{单组件贡献度讨论}

综合排名已给出变体间的总分差异，本节结合热力图与雷达图，按组件逐一给出失效场景与数值退化幅度，以界定每个组件的必要条件与适用范围。

关于C1时变SL投影，M1关闭C1后，在P1单行人横穿与P2行人加停车两个动态压力场景中失败，平均到达时间$T_{\mathrm{arrive}}$退化至$t_{\max}+\AblationFailPenalty$\,s的惩罚上限。此处结果需要与Baseline对照理解。Apollo原生的\texttt{IsStatic()}过滤会直接将动态行人移出路径阶段交给ST兜底，MIKU主干保留该动态障碍物但依赖C1的时变投影处理；一旦关闭C1，动态行人退化为$t=0$快照占据车道，路径阶段被错误约束至当前位置而强制减速等待。因此M1在P1/P2的失败反映的是MIKU选择“纳入路径阶段”后若缺失时变投影配合则会产生副作用，而不是Baseline中“完全不纳入”口径下的同等问题。C1单独存在但缺失时变机制时需要Apollo式的过滤兜底，这也印证了C1与C2/C3组级决策的协同必要性。

关于C2--C3组级决策的关键作用，M2关闭C2与C3后，在P4单车道封闭导流压力场景失败，$\bar v$ 仅 \AblationMTwoScFourVAvg\,m/s，$s_{end}$ 为 \AblationMTwoScFourSEnd\,m，仅占目标距离的 \AblationMTwoScFourSEndPct\%。原因在于该场景的\DenseConstructionTotalObstacles{}个静态障碍物构成单一大型连通分量，逐障碍物贪心决策时入口漏斗的\DenseConstructionEntranceCones{}锥每个独立选择左推，与维持段水马的右推产生$l_{\min} > l_{\max}$冲突，触发blocked-then-trim降级停车。组级最大间隙搜索将整个连通分量识别为单一组并赋予一致的左/右分配，消解了贪心决策互斥。

关于C4差异化裕度的关键作用，M3关闭C4后，在P3窄路交通锥压力场景失败，$\bar v$ 仅 \AblationMThreeScThreeVAvg\,m/s。该场景的双侧交通锥间隙宽度仅略大于车身宽度，统一额外缓冲$d_{buf}=\DeltaBaseline$\,m对小尺寸锥桶过度保守，将通行带压缩至$l_{\min} > l_{\max}$。多因子威胁度模型按障碍物类型与TTC动态调整额外缓冲$d_{buf,i}$至$[\DeltaIntervalLow, \DeltaIntervalHigh]$\,m区间，并由此得到差异化总安全裕度$\delta_i$，对低威胁锥桶下调缓冲，使本不可行的窄路通过变为可行。

关于C5走廊注入的架构价值，M4关闭C5后，在压力场景中的通过数为\AblationMFourPassCount{}/\AblationScenarioTotal，综合得分相对M5的差值仅为\AblationMarginMFourMFive 分，计算开销差异近似为零。这一近似持平的原因在于，当前压力场景均为单车道静态或低动态场景，路径阶段已通过C1的时变投影提前规避了所有动态时间冲突，速度阶段仅需保守通过，C5走廊约束未触发活跃边界。C5的架构意义不在于提升简单场景的指标，而在于为Apollo的路径-速度解耦架构补全了此前缺失的约束传递通道，将路径阶段对动态障碍物的时空决策以$(s_k,\tau_k)$形式注入速度QP，使路径规划与速度规划在动态场景下保持一致性。该通道在多动态障碍物时间窗交织或交叉路口冲突点决策等场景中将成为速度规划的必要约束源。当前压力场景的消融结果表明，C5在不被激活时对系统无实质负面影响，相关开销在毫秒量级，满足\PlannerCycleMs\,ms规划周期要求。

消融实验定量验证了C1、C2、C3、C4对MIKU算法的作用。C1与C2--C3共同决定动态和大型连通分量场景的通行成功率，C4决定窄路锥列下的可行性。C5在当前压力场景中未被激活，其作用需要在多动态障碍物时间窗交织或交叉路口冲突点决策等场景中进一步验证，本文覆盖的场景不足以对C5给出定量评价。从数据看，多障碍物路径规划的改进重心在路径阶段的时变与组级建模即C1至C4；C5负责将路径决策向速度约束正向传递，为路径-速度协调提供架构接口，但其定量贡献需要后续实验补充。

\subsection{威胁度权重灵敏度分析}
\label{sec:sensitivity_exp}

第\ref{chap:threat}章式\eqref{eq:threat_score}中的五个权重$w_1$至$w_5$由定性排序与典型场景微调确定，有必要检验其对数值扰动的鲁棒性。本节沿用P1--P4压力场景，从单因子扰动与综合随机扰动两个维度设计灵敏度实验。

单因子扰动实验中，逐一将每个权重因子放大或缩小\SensPerturbPct\%并重新归一化，检测障碍物间的威胁度排序是否发生翻转。场景一单行人仅含单障碍物，无需检测排序；场景二行人加停车和场景三\NarrowCmpConeCountPerSide{}个水马加\NarrowCmpConeCountPerSide{}个交通锥排序完全稳定（\SensTwoSingleFlipCnt{}/\SensSingleFactorTrialCount 次翻转）；场景四（\DenseConstructionTotalObstacles 个障碍物）出现\SensFourSingleFlipCnt{}/\SensSingleFactorTrialCount 次翻转，但均发生在威胁度极接近的同类交通锥之间（$\Delta\Theta < \SensThetaCloseThreshold$），不影响实际通行决策。

综合随机扰动实验中，对五个权重同时施加$\pm\SensPerturbPct\%$的随机扰动并进行\SensTrialCount 次蒙特卡洛实验，统计差异化安全裕度$\delta_i$的波动范围与排序翻转率。各场景中$\delta_i$的最大绝对偏差，场景一为\SensOneMcMaxAbsDev{}\,m，场景二为\SensTwoMcMaxAbsDev{}\,m，场景三为\SensThreeMcMaxAbsDev{}\,m，场景四为\SensFourMcMaxAbsDev{}\,m，全场景最大绝对偏差不超过\SensWorstAbsDev{}\,m。最大相对偏差在场景一为\SensOneMcMaxRelDev{}\%，场景二为\SensTwoMcMaxRelDev{}\%，场景三为\SensThreeMcMaxRelDev{}\%，场景四为\SensFourMcMaxRelDev{}\%，全场景相对偏差均小于\SensWorstRelDev{}\%。$\delta_i$变化超过\SensDeltaExceedThreshold\,m的概率为\SensFourMcExceedProb{}\%，排序翻转率在场景一至场景三中均为\SensOneMcFlipRate{}\%，场景四为\SensFourMcFlipRate{}\%。场景四的翻转集中在$\Theta$差异不足\SensThetaCloseThreshold 的同类交通锥之间，对应的$\Delta\delta_i < \SensDeltaCloseThreshold$\,m，对通行带宽度的影响可以忽略。

上述实验表明，权重的合理扰动不改变障碍物间的相对威胁等级，对通行带宽度的影响在工程容忍范围内。需注意的是，本文权重参数面向城市道路30--50\,km/h工况标定，在高速公路大速度差场景下需重新评估。

\subsection{综合对比分析}

将表\ref{tab:full_comparison}按指标拆分，可以分别从通行成功率、平均速度与轨迹平顺性三条线索理解MIKU相对Baseline的改进幅度。

在通行成功率方面，P1--P4压力场景中，Baseline通过\MPressureBaselinePassCount{}/\MPressureScenarioTotal{}，MIKU通过\MPressureMikuPassCount{}/\MPressureScenarioTotal{}，说明原始临界构型确实暴露了Baseline路径边界构建的失效边界。在C1--C4可比场景中，Baseline与MIKU均完成\MComparableBaselinePassCount{}/\MComparableScenarioTotal{}与\MComparableMikuPassCount{}/\MComparableScenarioTotal{}通过，因此可比组不再把通过率作为主结论，而用于比较同一可达前提下的轨迹质量、求解开销以及约束组织方式。

在平均速度方面，C1--C4可比场景中，Baseline平均巡航速度为\MComparableBaselineAvgV\,m/s，MIKU为\MComparableMikuAvgV\,m/s，两者处于同一量级。此时速度本身不再是主要论点，更有意义的是动态场景中的减速度与jerk控制，以及约束前置是否降低整体求解开销。

在轨迹平顺性方面，C1中Baseline与MIKU均能通过，但MIKU将纵向加速度峰值由\MFiveBaselineMaxAbsA\,m/s$^2$降至\MFiveMikuMaxAbsA\,m/s$^2$，jerk峰值由\MFiveBaselineMaxAbsJerk\,m/s$^3$降至\MFiveMikuMaxAbsJerk\,m/s$^3$；C2中对应峰值由\MSixBaselineMaxAbsA\,m/s$^2$与\MSixBaselineMaxAbsJerk\,m/s$^3$降至\MSixMikuMaxAbsA\,m/s$^2$与\MSixMikuMaxAbsJerk\,m/s$^3$。C3中MIKU的横向加速度峰值由\MSevenBaselineMaxAbsALat\,m/s$^2$降至\MSevenMikuMaxAbsALat\,m/s$^2$。这些结果说明复制场景不止用于记录通过与失败的对照，而是在双方可达时比较约束组织带来的轨迹质量差异。

MIKU新增的扫描线分组与时变投影步骤的时间复杂度为$O(n \log n)$，而Baseline的\texttt{PathBoundsDecider}在每个离散位置遍历所有障碍物，时间复杂度为$O(nK)$，其中$K$为路径离散点数，典型配置下$K \approx 200$。在当前\MScenarioTotal{}个场景的障碍物数量范围内，MIKU的分组与投影开销均在毫秒级，对总耗时影响较小。OSQP求解器本身对约束数量的增量不敏感，走廊约束以标准ST上界形式注入，不改变QP的稀疏结构，因此速度QP耗时差异主要来自约束前置后的问题规模变化。整体满足Apollo的\PlannerCycleMs\,ms规划周期要求。

\section{本章小结}

本章把MIKU集成至Apollo Planning的C++代码，在\texttt{LaneFollowPath}中新增MIKU路径边界分支，由\texttt{GetBoundaryFromMikuObstacles}替代原逐静态障碍物流程，并以\texttt{STDrivableBoundary}的形式把速度约束传入\texttt{PiecewiseJerkSpeedOptimizer}，下游Control模块接口保持不变。消融实验以P1--P4压力场景量化\AblationComponentTotal{}个组件的必要性，C1--C4可比场景通过微调时间窗或障碍物位置保证Baseline与MIKU均能通过，用于连续指标比较。Dreamview在环截图展示C++接入后在动态超车、窄路通行与单车道封闭导流中的规划行为。综合对比表明，压力场景中Baseline通过0/4，MIKU通过4/4；可比场景中两者均通过4/4，MIKU降低了动态场景的加速度与jerk峰值，C1--C4平均QP总耗时由13.58\,ms降至4.51\,ms。压力场景下Baseline进入FallbackPath降级分支，其耗时数据不构成效率比较依据。
